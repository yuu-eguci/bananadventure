# メッセージ画面タップで次へ進む

## タスク概要

選択肢を押したときに出るメッセージダイアログを、OK ボタン以外でも閉じて次へ進めるようにしたい。

### ユーザーフィードバック

「メッセージの OK が見えづらくて押しづらい」

### 今回の要望

- OK ボタンを押さなくても、画面のどこを押しても次へ進めるようにしたい
- 「どこを押しても大丈夫なメッセージ画面なんだ」とユーザーにわかるようにしたい

### 現状の実装

- `HomePage.tsx` の `Dialog` でメッセージを表示している
- `onCloseDialog` を `Dialog.onClose` と OK ボタンの両方に設定済み
- `DialogActions` 内に `Button OK` が配置されている

### 変更のゴール

1. **タップ/クリックで閉じる**: Dialog の外側 (Backdrop) をタップしても閉じる
2. **視覚的なヒント**: 「どこかタップして次へ」という旨をメッセージ画面内に表示する
3. OK ボタンは残すか廃止するかも含めて設計する

## 詳細設計（プランナー記入欄）

### 1. Dialog の「どこでもタップで閉じる」実装方針

今回は「Backdrop だけ」じゃなく、**メッセージ本体を含めてどこでもタップで進める**方針でいくよ。

- 前提として MUI v7 の `Dialog` には `disableBackdropClick` はないので使わない
- `Dialog` の `onClose` は残し、`reason` が `"backdropClick"` / `"escapeKeyDown"` のときに `onCloseDialog` を呼ぶ
- さらに `Dialog` 内側タップでも進めるため、`DialogContent`（または `slotProps.paper`）に `onClick` をつけて `onCloseDialog` を呼ぶ
- 連打で二重遷移しないように、`onCloseDialog` 側に「進行中フラグ」を入れて再入をブロックする

これで、ユーザー目線では「ダイアログが出ている間は、画面のどこを触っても次へ進む」体験になる。

### 2. ユーザーへの視覚的ヒント表示

「押せる場所が分からない」を防ぐため、ダイアログ下部にヒントを固定表示する。

- 表示文言: `タップして次へ`
- 表示位置: `DialogContentText` の下（`DialogContent` 内）
- 見た目: `Typography` の `caption` + `TouchApp` 系アイコンを横並び
- 動き: `opacity` を使ったゆるい点滅（例: 1.2s 無限ループ）で「次へ進める」ことを自然に伝える
- 補助: `DialogContent` 全体に `cursor: "pointer"` を付け、クリック可能な面だと分かるようにする

派手すぎる演出にはせず、読み終わったらすぐ次に進める軽い案内にする。

### 3. OK ボタンの扱い

今回は **OK ボタンは廃止** する。

- `DialogActions` ごと削除
- 理由1: 押下ターゲットを画面全体に広げる要望と完全一致する
- 理由2: 「OK を探して押す」認知コストをなくせる
- 理由3: モーダルの縦幅が少し減って、メッセージ本文とヒントが見やすくなる

閉じる導線は `Backdrop` / `Dialog 本体タップ` / `Esc` で確保する。

### 4. 変更対象ファイル一覧

- `webapp/frontend-react/src/pages/HomePage.tsx`
  - `Dialog` の close ハンドリング更新（`onClose` の `reason` と本体タップ対応）
  - 進行中フラグの追加（連打ガード）
  - ヒント UI（テキスト + アイコン + 点滅）追加
  - `DialogActions` と `OK` ボタン削除
- `docs/agent/028-メッセージ画面タップで次へ進む.md`
  - 本詳細設計の追記

### ✅ 良かった点

- **Dialog タップで閉じる実装方針**: MUI の仕様とちゃんと合ってる! `onClose` の `reason` チェックは正しいアプローチ。
- **連打ガード**: 進行中フラグでの再入ブロックを考慮済み。二重遷移を防ぐ観点がしっかりしてる。
- **OK ボタン廃止**: 要望に完全一致してるし、認知コスト削減の理由が明確。縦幅削減も副次効果として Good。

### ⚠️ 指摘事項

#### 1. `onClose` の `reason` ハンドリングが不完全

プランでは「`reason` が `"backdropClick"` / `"escapeKeyDown"` のときに `onCloseDialog` を呼ぶ」って書いてるけど、現状の実装 (`HomePage.tsx:457`) では `onClose={onCloseDialog}` と無条件で渡してるから、すでに全部の close イベントで `onCloseDialog` が呼ばれてる状態。

- **問題**: 「`reason` をチェックする」方針と「現状の実装」が乖離してる。もし本当に `reason` をチェックしたいなら、`onClose` を `(event, reason) => { if (reason === ...) onCloseDialog(); }` みたいに書き直す必要がある。
- **提案**: 実は現状のままで要件は満たせてるから、設計書の文言を「`Dialog.onClose` に `onCloseDialog` を渡し、Backdrop / Esc のどちらでも閉じられるようにする」程度に修正したほうが実装とズレがなくて良さそう。

#### 2. Dialog 本体クリックの具体的な実装箇所があいまい

「`DialogContent` または `slotProps.paper` に `onClick` をつける」って書いてあるけど、どっちにつけるかで体験が変わる:

- `DialogContent` につけた場合: Content 部分だけがクリック可能
- `slotProps.paper` につけた場合: Dialog 全体 (余白含む) がクリック可能

**提案**: 「画面のどこを押しても次へ進める」という要望を忠実に実現するなら、`slotProps.paper` に `onClick` をつける方針を明記したほうがいい。ただし、これをやると「ダイアログ本文を選択しようとしたら閉じちゃった」リスクもあるので、以下も併記すべき:

- テキスト選択を許可するなら `onMouseDown` を使って「クリックのみで発火」させる
- または `DialogContent` には `onMouseDown={(e) => e.stopPropagation()}` をつけて、本文選択を許容しつつ、余白クリックで閉じる設計にする

#### 3. 連打ガードの実装詳細が不足

「`onCloseDialog` 側に進行中フラグを入れる」とあるけど、具体的に:

- フラグの名前 (`isClosing` とか) は?
- `useRef` で持つのか `useState` で持つのか?
- どのタイミングでフラグを `true` にして、どこで `false` に戻すのか?

現状の `onCloseDialog` (94-114行目) を見ると、すでに `setJingleOpen(true)` で Jingle Backdrop が出てるから、ある程度は連打がブロックされるかも。でも、厳密には:

```typescript
const isClosingRef = useRef(false);

const onCloseDialog = async () => {
  if (isClosingRef.current) return; // 再入ガード
  isClosingRef.current = true;

  // ... 既存の処理 ...

  setTimeout(() => {
    // ...
    isClosingRef.current = false; // 解除
  }, 1000);
};
```

みたいな実装が必要だよね。設計書にこの程度のコード例か擬似コードを書いておいたほうが、実装者が迷わない。

#### 4. 視覚ヒントのアイコン名が曖昧

「`TouchApp` 系アイコン」って書いてあるけど、MUI に `TouchApp` アイコンはないｗ 多分 `@mui/icons-material` の `TouchAppOutlined` とか `TouchApp` (旧バージョン) を指してると思うけど、最新の MUI では名前が変わってたり廃止されてる可能性もある。

**提案**: `TouchAppOutlined` または `GestureOutlined` / `PanToolOutlined` など、具体的なアイコン名を明記する。実装時に探す手間がなくなる。

#### 5. 点滅アニメーションの CSS 実装方法が未記載

「`opacity` を使ったゆるい点滅 (1.2s 無限ループ)」とあるけど、MUI の `sx` で `@keyframes` を定義するのか、`styled` を使うのか、グローバル CSS に書くのか、実装方法が書いてない。

**提案**: せめて「`sx` の `animation` プロパティで `keyframes` を inline 定義」みたいに方針を書いておくといいかも。

#### 6. 変更対象ファイルの網羅性

現状は `HomePage.tsx` と設計書だけだけど、以下も検討したほうがいいかも:

- **i18n 対応**: 「タップして次へ」のテキストはハードコードでいいのか? 本プロジェクトは i18next が入ってるから、翻訳キーを追加する必要があるかもね。
- **型定義の追加**: 新しく `useRef<boolean>` を使うなら、特に型定義ファイルの変更は不要だけど、もし共通化するなら hooks ファイルを追加するかもしれない。

**提案**: i18n 対応が必要なら、`webapp/frontend-react/src/locales/en/translation.json` (またはそれに相当するファイル) を変更対象に追加する。不要なら「今回はハードコードで問題なし」と明記しておく。
