# ストロングゼロ効果変更とアイテム使用モーダル

## タスク概要

ユーザからの以下の feedback に対応する。

### ユーザーフィードバック

1. ストロングゼロアイテムの効果をバナナメーター -9 にしてほしい
2. アイテムをタップしたときにメッセージモーダルを表示してほしい
   - 「(アイテム名) を使用した！」
   - 「(ここに効果)」

### 今回の要望

- ストロングゼロの `bananaMeterEffect` を -9 に変更する
- アイテムタップ時に、028 と同様のモーダルで使用結果メッセージを表示する

## 詳細設計（プランナー記入欄）

### 変更のゴール

1. ストロングゼロ使用時の効果を、プレイヤーのばななメーター `-9` に統一する
2. アイテムボタン押下時に、028 と同じ操作感（どこタップでも閉じる）のメッセージモーダルを表示する
3. 既存の「シーン選択用モーダル」と「アイテム使用モーダル」を衝突させない

### 現状整理（実装前提）

- `HomePage.tsx` の `Dialog` は、現状 `onCloseDialog` でシーン遷移処理まで実行している
- そのため、アイテム使用で同じ close ハンドラをそのまま使うと、意図しないシーン遷移が混ざるリスクがある
- `SceneService.useItem` は `Item.bananaMeterDelta` を参照してメーター加減算している
- 現在のストロングゼロ JSON は `bananaMeterDelta: 0` / `bananaMeterDeltaPercent: -9` なので、実装上は効果が 0 扱いになっている

### 1. ストロングゼロ効果値の変更方針

現行モデルでは `bananaMeterEffect` というフィールドは使っていないので、今回の「効果 -9」は `bananaMeterDelta = -9` として反映する。

- 変更対象: `webapp/frontend-react/src/data/bananadventure-scenes.json`
- 対象アイテム: `id: 1`（ストロングゼロ）
- 変更内容:
  - `bananaMeterDelta: 0` → `bananaMeterDelta: -9`
  - `bananaMeterDeltaPercent` は現状未使用のため、今回仕様では参照しない

これで `SceneService.useItem` の既存ロジックだけで、使用時に確実に `-9` が反映される。

### 2. アイテム使用モーダルの状態設計

シーン用とアイテム用で close 後の挙動が違うので、モーダル種別を明示的に管理する。

- 追加 state（`HomePage.tsx`）
  - `dialogType: "scene" | "item" | null`
  - 既存の `openDialog` / `dialogMessage` は流用
- 開くときのルール
  - シーン選択時: `dialogType = "scene"` で開く
  - アイテム使用時: `dialogType = "item"` で開く
- 閉じるときのルール
  - `scene`: 既存どおりシーン遷移を実行
  - `item`: 遷移は実行せず、モーダルを閉じるだけ

この分離で、028 の UX（どこでもタップで閉じる）を維持しながら、挙動の混線を防ぐ。

### 3. アイテム押下フロー詳細

`onPressItem` を「効果適用→モーダル表示」まで含むフローにする。

1. `viewModel` がなければ return
2. `service.useItem({ viewModel, itemId: item.id })` を実行して `setViewModel` 反映
3. モーダル表示文言を作る
   - 1 行目: `{item.text} を使用した！`
   - 2 行目: 効果文（例: `ばななメーター -9`）
4. `dialogType = "item"` をセットして `openDialog = true`

補足:
- 押下ボタンは既存どおり `item.used` で `disabled` のため、使用済みアイテムは再表示されない
- `SceneService.useItem` 側のガード（未所持/使用済み）と UI 側の `disabled` で二重防御する

### 4. モーダル UI の実装方針（028 準拠）

- 見た目と操作は 028 と同様
  - `Dialog.onClose` で close
  - `slotProps.paper.onClick` で本文タップでも close
  - ヒント文（例: `どこでもタップして次へ`）は既存実装をそのまま共用
- アイテムモーダルの本文は 2 行表示にする
  - 1 行目: 使用通知
  - 2 行目: 効果通知

### 5. 変更対象ファイル

- `webapp/frontend-react/src/data/bananadventure-scenes.json`
  - ストロングゼロの `bananaMeterDelta` を `-9` へ変更
- `webapp/frontend-react/src/pages/HomePage.tsx`
  - `dialogType` 追加
  - `onCloseDialog` をモーダル種別で分岐
  - `onPressItem` のメッセージ作成・モーダル表示追加
  - 既存 Dialog は 028 の操作感を維持
- `docs/agent/029-ストロングゼロ効果変更とアイテム使用モーダル.md`
  - 本詳細設計の追記

### 受け入れ条件

- ストロングゼロを使用すると、ばななメーターが `-9` される
- アイテムボタン押下時にモーダルが開き、以下 2 行が表示される
  - `(アイテム名) を使用した！`
  - `(効果文)`
- モーダルは 028 と同様に、どこをタップしても閉じられる
- アイテムモーダルを閉じてもシーン遷移は発生しない
- 既存のシーン選択モーダルの遷移挙動は壊れない

### 非対象（今回やらないこと）

- アイテム効果文の i18n 化（現状 UI 方針に合わせて日本語ハードコード維持）
- `bananaMeterDeltaPercent` を使った割合計算ロジック追加
- 新規の演出（SE、専用アニメーション、専用アイコン）追加

### 1. ストロングゼロ効果値の変更方針について

**【指摘 1-1】 JSON の該当箇所の特定が不正確**

設計書には「対象アイテム: `id: 1` (ストロングゼロ)」と書かれているけど、 Grep 結果を見る限り **`id: 1` のアイテムはストロングゼロだけではない**。

JSON では `itemsOnSelect` の中に以下のようにストロングゼロが記述されている (scene 8 内の sceneChoice 2 の中)：

```json
{
  "id": 1,
  "text": "ストロングゼロ",
  "description": "",
  "responseText": "",
  "bananaMeterDelta": 0,
  "bananaMeterDeltaPercent": -9,
  "image": "/item-image/item-01-lemon-strong-zero.webp",
  "used": false
}
```

この JSON 構造だと、 `id: 1` はストロングゼロ **アイテム** で間違いないけど、 **他にも複数の `id: 1` が存在していて、それらは sceneChoice や scene の `id` として使われている**。

設計書には **「webapp/frontend-react/src/data/bananadventure-scenes.json の scene 8 の sceneChoices[1] の itemsOnSelect[0]」** みたいに、もっと明確に対象箇所を指定してほしいｗ

**【指摘 1-2】 Item の `bananaMeterDelta` 変更がパーマネントかどうかの説明不足**

今回、 JSON の中にあるストロングゼロアイテムの定義を `bananaMeterDelta: -9` に書き換えるわけだけど、この値は **初回取得時の初期値**。プレイヤーがこのアイテムを既に取得済みの状態で、リロードした場合やセーブデータをロードした場合に、この変更が適用されるかどうかが不明瞭。

現状の実装では、 Player のアイテムは JSON から読み込んだ sceneChoice の `itemsOnSelect` を直接 push しているだけだから、 **既に Player が持っているアイテムのデータには JSON 変更が反映されない**。

ただし、現状は完全にフロントエンドで完結していてセーブ機能もないから、リロードすれば常に初期状態から始まるので問題ないとも言える。

この点について「現状の仕様ではセーブ機能がないため、リロード時には常に初期状態から始まるので問題ない」といった **前提条件を明記** するべきｗ

### 2. アイテム使用モーダルの状態設計について

**【指摘 2-1】 `dialogType` の初期値が不明確**

設計書には `dialogType: "scene" | "item" | null` という state を追加すると書いてあるけど、 **初期値が何か** が書かれていない。

既存の実装を見る限り、 Dialog は `openDialog` が `false` のときは閉じているので、初期値は `null` で良いと思うけど、それを明記してほしいｗ

**【指摘 2-2】 `selectedSceneChoice.current` の扱いが不明瞭**

現状の実装では、 `selectedSceneChoice.current` という ref を使ってシーン選択時に選ばれた choice を保持しているけど、設計書には **アイテム使用時にこの ref をどう扱うか** が書かれていない。

アイテム使用時には `selectedSceneChoice.current` を `null` のまま、もしくはそのまま触らない方針なのか？ それとも何か別の値をセットするのか？

現状のコードを見る限り、 `onCloseDialog` の中で `selectedSceneChoice.current` が `null` かどうかをチェックしているから、 **アイテム使用時には `selectedSceneChoice.current` を触らず、 `dialogType` で分岐することで問題を回避する** という設計だと推測できるけど、それを **明示的に書いてほしい**。

### 3. アイテム押下フロー詳細について

**【指摘 3-1】 効果文の生成ロジックが不明確**

設計書には「効果文 (例: `ばななメーター -9`)」と書いてあるけど、 **どうやってこの文字列を生成するのか** が書かれていない。

`Item` インターフェースを見ると、 `bananaMeterDelta` というフィールドがあるから、これを使って `ばななメーター ${item.bananaMeterDelta}` みたいに生成するのか？

それとも、 `Item.description` や `Item.responseText` を使うのか？

現状の JSON を見ると、ストロングゼロの `description` と `responseText` はどちらも空文字列 (`""`) なので、 **`bananaMeterDelta` をベースに動的生成する** しかないと思うけど、それを明記してほしいｗ

また、 `bananaMeterDelta` が正の値の場合と負の値の場合で、表示を変えるのか (例: `-9` なら `ばななメーター -9` 、 `+9` なら `ばななメーター +9`) とか、そういう細かい仕様も書いてほしい。

**【指摘 3-2】 モーダル表示のタイミングが曖昧**

設計書には「2. `service.useItem(...)` を実行して `setViewModel` 反映」と書いてあるけど、 **`setViewModel` が完了してから `openDialog` をセットするのか、それとも並列でやるのか** が不明瞭。

現状の `onPressSceneChoice` を見ると、 `setDialogMessage` → `setOpenDialog(true)` という順序で同期的にやっているから、それと同じ方式で良いと思うけど、それを明記してほしいｗ

**【指摘 3-3】 `useItem` の戻り値を受け取るタイミングでの状態の整合性チェックが不足**

`SceneService.useItem` の実装を見ると、 **バナナメーターが 0 以下になった場合、ゲームオーバーシーンに遷移する** 処理が入っている (SceneService.ts:142-143)。

つまり、ストロングゼロを使ってバナナメーターが 0 以下になった場合、 **`useItem` の戻り値として返される `SceneViewModel` の `scene` はゲームオーバーシーン (id: 15) になっている** 可能性がある。

このとき、モーダルに「ストロングゼロを使用した！ / ばななメーター -9」と表示されるのは良いけど、 **モーダルを閉じた後にゲームオーバーシーンが表示されるべきかどうか** が設計書に書かれていない。

おそらく、モーダルを閉じた後にゲームオーバーシーンが表示される挙動で問題ないと思うけど、それを **受け入れ条件** に明記してほしいｗ

また、 `useItem` の実装を見ると、 `resolveTriggeredScene` も呼ばれているから、 **アイテム使用後に triggerItems によってシーン遷移が発生する可能性もある** (SceneService.ts:145-150)。

これも同様に、モーダルを閉じた後にシーンが変わっていても問題ないか？ それとも、アイテム使用時には triggerItems は発火させない設計にするのか？

この辺りの **エッジケースの挙動** を明確にしてほしい。

### 4. モーダル UI の実装方針について

設計書には「ヒント文 (例: `どこでもタップして次へ`) は既存実装をそのまま共用」と書いてあるけど、アイテムモーダルでも **このヒント文を表示するのか** が明確ではない。

028 のシーン選択モーダルと同様の UX にするなら、ヒント文も表示するべきだと思うけど、それを明記してほしいｗ

**【指摘 4-2】 モーダル本文の改行処理が不明確**

設計書には「1 行目: 使用通知 / 2 行目: 効果通知」と書いてあるけど、 **実際の React コードでどうやって改行するのか** が書かれていない。

既存の `onPressSceneChoice` を見ると、 `setDialogMessage(responseText)` という形で `string` を渡しているから、改行を含めるなら `{item.text} を使用した！\nばななメーター ${item.bananaMeterDelta}` みたいな文字列を作るのか？

それとも、 JSX で `<Box>` を 2 つ並べるのか？

現状の `DialogContentText` は単一の children を受け取っているから、 **文字列に `\n` を含めても MUI の `Typography` では改行されない** 可能性がある。

改行するには `white-space: pre-line` のような CSS を追加するか、 JSX で複数の要素に分けるかのどちらかが必要だけど、それを明記してほしいｗ

つまり、モーダルが開いている最中に、背景のバナナメーター表示は既に `-9` された値になっていると思うけど、それを明記してほしいｗ

### 6. その他の懸念点

**【指摘 6-1】 `onPressItem` の非同期処理の扱いが不明**

現状の `onPressItem` は `async` 関数だけど、設計書には **`await` を使うのか、それとも `void` で fire-and-forget するのか** が書かれていない。

既存の `onPressSceneChoice` を見ると、 `CardActionArea` の `onClick` で `() => onPressSceneChoice(choice)` という形で呼んでいて、 `await` はしていない。

同様に、アイテム使用時も `onClick={() => onPressItem(item)}` という形で呼ぶなら、 `await` はしないことになるけど、それで問題ないか？

`SceneService.useItem` の中では `getScene` や `resolveTriggeredScene` が `await` されているから、非同期処理が完了する前にモーダルが開いてしまう可能性があるけど、それで大丈夫か？

この辺りの **非同期処理の順序** を明確にしてほしいｗ

**【指摘 6-2】 ゲームオーバー時のモーダル表示が不適切な可能性**

【指摘 3-3】 でも触れたけど、アイテム使用後にバナナメーターが 0 以下になった場合、 `useItem` の戻り値として返される `SceneViewModel` の `scene` はゲームオーバーシーン (id: 15) になる。

このとき、 **モーダルに「ストロングゼロを使用した！ / ばななメーター -9」と表示されて、モーダルを閉じるとゲームオーバーシーンが表示される** という挙動になると思うけど、 **ユーザー体験として適切か？**

もしかしたら、モーダルの中で「ばななメーターが 0 以下になりました！」みたいな追加メッセージを表示した方が親切かもしれないｗ

この辺りは設計の範囲外かもしれないけど、一応指摘しておくね。
