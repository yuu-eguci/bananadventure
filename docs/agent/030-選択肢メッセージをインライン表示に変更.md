# 選択肢メッセージをインライン表示に変更

## タスク概要

### ユーザーフィードバック

- 選択肢ボタン → モーダル → (暗くなる) → 選択肢 の流れで視線がいったりきたりして UX が悪い
- モーダルで暗くなるのが目に悪い

### 今回の要望

- 選択肢ボタンが並んでいるエリアに、選択後のメッセージを表示する（モーダルを使わない）
- 暗くしない（Backdrop/Dialog なし）
- 「タップして次へ」の機能はそのまま維持する

### 提案されているアプローチ（ユーザー案）

- 選択肢が並んでいる場所に、そのまま選択後のメッセージを出す
- 普通のゲームと同じように暗転しない
- よりよい案があればそちらを採用して OK

## 詳細設計（プランナー記入欄）

### 変更のゴール

1. シーン選択後のメッセージを `Dialog` ではなく、選択肢エリア内にインライン表示する
2. 画面暗転（`Backdrop` + `Dialog`）を使わない
3. 028 で導入した「タップして次へ」の操作感は維持する
4. 029 のアイテム使用モーダルも同じ思想でインライン化し、メッセージ表示体験を統一する

### 現状実装の整理（HomePage.tsx 前提）

- 現在は `openDialog` / `dialogMessage` / `dialogType` で、シーン選択メッセージとアイテム使用メッセージを同じ `Dialog` で表示している
- `onCloseDialog` は `dialogType` で `scene` / `item` を分岐している
- シーン選択時は `selectedSceneChoice.current` を保持して、close 時に `selectSceneChoice` を実行している
- アイテム使用時は `service.useItem` 済みの結果を `Dialog` で通知し、close 時は遷移せず閉じるだけ

いまの構成は機能としては成立しているけど、要望どおり「暗転なし」にするには `Dialog` 依存を外すのが最短。

### 1. 状態設計の変更方針

`Dialog` 専用 state をやめて、インラインメッセージ state に置き換える。

- 削除する state / ref
  - `openDialog`
  - `dialogMessage`
  - `dialogType`
  - `selectedSceneChoice`
- 追加する state（案）

```ts
type InlineMessageState =
  | { kind: "scene"; sceneChoiceId: number; text: string }
  | { kind: "item"; itemName: string; signedDeltaText: string }
  | null;

const [inlineMessage, setInlineMessage] = useState<InlineMessageState>(null);
```

- 継続利用する ref
  - `isClosingRef` は流用（必要なら `isAdvancingRef` にリネーム）し、連打による二重遷移を防ぐ

### 2. シーン選択フロー（インライン化）

`onPressSceneChoice` は次の分岐にする。

1. `viewModel` が `null`、または `isClosingRef.current === true` のときは return
2. `choice.responseText` が空なら、即 `advanceScene(choice.id)` を実行
3. `choice.responseText` があるなら、`inlineMessage = { kind: "scene", sceneChoiceId, text }` をセット

`advanceScene`（新規関数）で、現行 `onCloseDialog(scene)` の遷移処理を引き継ぐ。

- `setJingleOpen(true)` → `service.selectSceneChoice(...)` → `setViewModel(...)`
- 1 秒後に `setJingleOpen(false)` と `setInlineMessage(null)` を実行
- 例外時は `isClosingRef.current` を戻して復帰

### 3. インライン表示 UI（選択肢エリア）

表示位置は「選択肢ボタンが並んでいるエリア」をそのまま使う。

- `inlineMessage?.kind === "scene"` の間は、選択肢カード群を一時的に隠して、メッセージカードを表示
- メッセージカード全体を `CardActionArea` でタップ可能にし、タップで `advanceScene(inlineMessage.sceneChoiceId)` を実行
- 下部ヒントは 028 の意図を維持して `TouchAppOutlined + タップして次へ` を表示
- 既存 `tapHintBlink` アニメーションは再利用する

これで「視線移動なし」「暗転なし」「タップで次へ」を同時に満たせる。

### 4. アイテム使用メッセージ（029）の扱い

今回は **アイテム使用メッセージもインライン化する** 方針にする。

理由:

- ユーザーフィードバックの本質が「暗転がつらい」なので、シーンだけ直すと体験が不統一になる
- `Dialog` を残すと状態分岐が複雑に残り続ける
- 既存の `dialogType` 分岐を外して責務を分離できる

アイテム側のフロー:

1. `service.useItem` 実行
2. `setViewModel(updatedViewModel)` 実行
3. `signedDeltaText` を生成（正数は `+` を付ける）
4. `inlineMessage = { kind: "item", itemName, signedDeltaText }` をセット

表示位置:

- 右カラムのインベントリ見出しの直下に、アイテム用インラインメッセージカードを表示
- カード全体タップで `setInlineMessage(null)`（シーン遷移はしない）

### 5. 操作衝突の制御

メッセージ表示中に別操作が入ると状態が混線するので、以下で制御する。

- `inlineMessage?.kind === "scene"` 中
  - 選択肢ボタンは非表示（メッセージカードへ置換）
  - アイテムボタンは `disabled` 扱いにする
- `inlineMessage?.kind === "item"` 中
  - アイテムメッセージカードのみ表示し、同時に他アイテム押下は不可
  - シーン選択ボタンも `disabled` にして一貫性を保つ

### 6. 実装対象ファイル

- `webapp/frontend-react/src/pages/HomePage.tsx`
  - `Dialog` 依存の state / UI / close ハンドラを削除
  - `inlineMessage` state と `advanceScene` を追加
  - 選択肢エリアのインラインメッセージ UI 追加
  - インベントリエリアのアイテム用インラインメッセージ UI 追加
  - タップヒント UI は既存アニメーションを流用
- `docs/agent/030-選択肢メッセージをインライン表示に変更.md`
  - 本詳細設計とレビュー追記

### 7. 受け入れ条件

- シーン選択で `responseText` がある場合、選択肢エリアにメッセージがインライン表示される
- インラインメッセージカードのタップで次シーンへ進む
- シーン選択時に `Dialog` / `Backdrop` は表示されない
- `responseText` が空の選択肢は、従来どおり即遷移する
- アイテム使用時はモーダルではなくインベントリ側のインラインメッセージで結果を表示する
- アイテムメッセージを閉じてもシーン遷移は発生しない
- 連打してもシーン遷移が二重実行されない

### 8. 非対象（今回やらないこと）

- メッセージ文言の i18n 化
- 画面全体タップでの遷移（今回はメッセージカード領域タップに限定）
- Jingle 演出自体の仕様変更

今回の要望（選択肢をインライン表示、暗転なし、タップで次へ維持）に対して、実装時の迷いどころを潰せる粒度まで設計できている。

### 4. Claude Code によるレビュー

#### 4.1 設計の評価点

1. **Dialog 削除方針は妥当**
   現状の `Dialog` (`HomePage.tsx:508-545`) と `Backdrop` (`HomePage.tsx:485-496`) を削除してインライン表示に切り替える方針は、UX 要望の「暗転がつらい」「視線移動がつらい」に対する最短解になってる。`openDialog` / `dialogMessage` / `dialogType` の state 三点セットを捨てて、判別可能な `InlineMessageState` の tagged union に統一するのはクリーンな構成だと思うよ。

2. **操作衝突制御の方針が明確**
   「セクション 5」で、メッセージ表示中は他操作を `disabled` にする方針を明記してる。これは `isClosingRef.current` と組み合わせて、連打や二重遷移を防ぐための必須制御。現状の `onCloseDialog` (`HomePage.tsx:108-156`) が `isClosingRef` で早期 return してるのを引き継げば問題ない。

3. **アイテムもインライン化して体験を統一**
   029 で追加されたアイテム使用モーダルを残すと「シーンは暗転なし、アイテムは暗転あり」で UX が不統一になる。設計書が「アイテム側もインライン化」を選んだのは一貫性があって良いね。

#### 4.2 指摘事項と改善案

##### 指摘 1: `advanceScene` の処理タイミングとインラインメッセージのクリア順序が不明確

**問題点:**
セクション 2 では「`advanceScene` で 1 秒後に `setJingleOpen(false)` と `setInlineMessage(null)` を実行」とあるけど、この順序だと「シーン遷移が完了してから 1 秒後にメッセージを消す」のか「ジングルが終わった瞬間にメッセージを消す」のかが曖昧。

また、ユーザーがメッセージカードをタップしたときに `advanceScene` を実行するわけだけど、タップした瞬間にメッセージカードを消すのか、それともジングルが終わるまでメッセージカードを残すのか、どっちが意図か書かれてない。

**改善案:**
以下のどちらかを設計書に明記すべき。

- 案 A: タップ時に即座に `setInlineMessage(null)` を実行し、メッセージカードを消してから `setJingleOpen(true)` で遷移開始
- 案 B: タップ時はメッセージカードを残したまま `setJingleOpen(true)` で遷移開始し、1 秒後の `setTimeout` で `setInlineMessage(null)` と `setJingleOpen(false)` を同時実行

あたしとしては、**案 B** を推奨するよ。理由は以下の通り:

1. 案 A だとメッセージカードが消えてから選択肢ボタンが一瞬復活して、その後ジングルが開始される流れになって、視覚的にチラつきが出る
2. 案 B ならメッセージカードを表示したままジングルに入り、ジングルが終わってから選択肢が復活するので、視線移動が最小限で済む
3. 現状の `Dialog` では「タップ → ジングル開始 → ジングル終了 → Dialog 閉じる → 選択肢復活」の流れなので、案 B はこれに近い

ただし案 B だと、`inlineMessage?.kind === "scene"` のときに選択肢ボタンを非表示にする処理と、ジングル中に選択肢ボタンを非表示にする処理が重複する可能性がある。この辺の制御をどうするか明記してほしい。

##### 指摘 2: アイテムメッセージの表示位置が「右カラムのインベントリ見出しの直下」だと視線誘導が悪い可能性

**問題点:**
セクション 4 で「右カラムのインベントリ見出しの直下に、アイテム用インラインメッセージカードを表示」とあるけど、これだと以下の問題がある:

1. インベントリ見出しの直下だと、アイテムボタンの上に被る形になる。アイテムが縦 2 列で並んでいる場合、メッセージカードが上段のアイテムを隠してしまうと「どのアイテムを押したっけ？」が分かりにくくなる
2. 現状の `Dialog` はアイテム名 + バナナメーター増減を 2 行で表示してる (`HomePage.tsx:169-172`) けど、これを右カラム内に表示すると、カラム幅が狭い場合にメッセージが折り返して縦に長くなり、下のアイテムが隠れる可能性がある

**改善案:**
以下のどちらかを検討してほしい:

- 案 A: アイテムメッセージは「アイテムボタンの上に重ねて表示」ではなく、「アイテムボタンを一時的に隠して、その場所にメッセージカードを表示」する。つまり、選択肢メッセージと同じように、アイテムボタン群を `display: none` にしてメッセージカードに置換する
- 案 B: アイテムメッセージは「インベントリ見出しの直下」ではなく、「バナナメーターカードの直下」に表示する。こうすると、アイテムボタンを隠さずに済む

あたしとしては、**案 A** を推奨するよ。理由は以下の通り:

1. 案 A ならシーン選択メッセージと同じ UI パターン (ボタン群をメッセージカードに置換) になって、実装も統一できる
2. 案 B だと「バナナメーターカードの直下」にメッセージカードが出現するけど、ユーザーの視線はアイテムボタンを押した場所にあるので、視線移動が発生してしまう

ただし案 A だと、アイテム使用後にメッセージを閉じたとき、アイテムボタンが復活する際に「どのアイテムが使用済みになったか」が一瞬分かりにくくなる可能性がある。この辺の UX をどう考えるか明記してほしい。

##### 指摘 3: `isClosingRef` のリネームは不要

**問題点:**
セクション 1 で「`isClosingRef` は流用（必要なら `isAdvancingRef` にリネーム）」とあるけど、リネームするメリットが不明確。

**理由:**
現状の `isClosingRef` は「Dialog を閉じる処理中かどうか」を表す名前だけど、本質的には「シーン遷移処理中かどうか」を表すフラグ。今回 `Dialog` を削除するからといって、このフラグの役割は変わらない。リネームすると以下のデメリットがある:

1. 既存コードとの diff が大きくなる
2. `isAdvancingRef` だと「何に対して advancing なのか」が不明確 (シーンなのか、メッセージなのか)
3. リネームしなくても「処理中フラグ」として機能するので、敢えて変える必要がない

**改善案:**
`isClosingRef` のままで良いと思う。どうしてもリネームしたいなら `isTransitioningRef` とかの方が意味が明確。

##### 指摘 4: `responseText` が空のときの即遷移ロジックで `isClosingRef` の制御が漏れる可能性

**問題点:**
セクション 2 で「`choice.responseText` が空なら、即 `advanceScene(choice.id)` を実行」とあるけど、現状の `onPressSceneChoice` (`HomePage.tsx:91-106`) では `responseText` が空のときに `onCloseDialog("scene")` を直接呼んでいて、この中で `isClosingRef` を制御してる。

新設計では `advanceScene` を呼ぶことになるけど、`advanceScene` が `isClosingRef` を制御する前提なのか、`onPressSceneChoice` が制御する前提なのかが不明確。もし `advanceScene` が制御する前提なら、`onPressSceneChoice` の冒頭で `isClosingRef.current === true` をチェックしてるのに、`responseText` が空のときにそのまま `advanceScene` を呼ぶと二重チェックになって冗長。

**改善案:**
以下のどちらかを設計書に明記すべき:

- 案 A: `onPressSceneChoice` の冒頭で `isClosingRef` をチェックして early return し、その後は `advanceScene` 内で `isClosingRef` を制御する
- 案 B: `onPressSceneChoice` では `isClosingRef` をチェックせず、`advanceScene` の冒頭で `isClosingRef` をチェックして early return する

あたしとしては、**案 A** を推奨するよ。理由は以下の通り:

1. `onPressSceneChoice` はユーザーの入力を受け付けるエントリーポイントなので、ここで早期 return する方が分かりやすい
2. `advanceScene` は「シーン遷移処理」に専念する関数なので、入力検証ロジックを混ぜない方が責務が明確

##### 指摘 5: セクション 5 の操作衝突制御で「シーン選択ボタンも disabled にする」は過剰制御の可能性

**問題点:**
セクション 5 で「`inlineMessage?.kind === "item"` 中はシーン選択ボタンも `disabled` にして一貫性を保つ」とあるけど、これは過剰制御じゃないかな。

**理由:**
アイテム使用は「シーン遷移しない」操作なので、アイテムメッセージを表示中でもシーン選択ボタンを押せる方が UX として自然。現状の `Dialog` 実装でも、アイテム使用ダイアログを開いてる最中に選択肢ボタンを押すことは `Dialog` の `Backdrop` がブロックしてるだけで、閉じればすぐ押せる。

今回の設計では `Dialog` を削除するので、アイテムメッセージを表示中でもシーン選択ボタンを押せるようにした方が、ユーザーが「メッセージを閉じる」操作を省略できて便利。

**改善案:**
以下のどちらかを検討してほしい:

- 案 A: `inlineMessage?.kind === "item"` 中は、シーン選択ボタンを `disabled` にせず、押せるようにする。アイテムメッセージは自動的に消える (シーン遷移が優先される)
- 案 B: 現設計のまま、シーン選択ボタンを `disabled` にする。ユーザーは「アイテムメッセージをタップして閉じる」という明示的な操作をしてから、次の選択肢を押す

あたしとしては、**案 A** を推奨するけど、これはプロダクトの UX ポリシー次第だから、ユーザーと相談して決めてほしい。もし案 A を採用するなら、「シーン選択ボタンを押したときに `inlineMessage` を `null` にクリアする」ロジックを `onPressSceneChoice` に追加する必要がある。

##### 指摘 6: 受け入れ条件に「アイテムメッセージ表示中にシーン遷移できるか」の記載がない

**問題点:**
セクション 7 の受け入れ条件に「アイテムメッセージを閉じてもシーン遷移は発生しない」とあるけど、「アイテムメッセージ表示中にシーン選択ボタンを押したらどうなるか」が書かれてない。これは指摘 5 と関連する。

**改善案:**
受け入れ条件に以下を追加すべき:

- アイテムメッセージ表示中にシーン選択ボタンを押した場合の挙動を明記する (disabled で押せないのか、押せるけどメッセージが消えて遷移するのか)

#### 4.3 レビュー結論

設計の大枠は良いけど、以下の点を明確にしてから実装着手すべき:

1. **必須**: `advanceScene` のタイミングとメッセージカードのクリア順序を明記する (指摘 1)
2. **必須**: アイテムメッセージの表示位置と UI パターンを再検討する (指摘 2)
3. **必須**: `isClosingRef` の制御タイミングを明記する (指摘 4)
4. **推奨**: アイテムメッセージ表示中のシーン選択ボタンの挙動を決める (指摘 5, 6)
5. **オプション**: `isClosingRef` のリネームは不要と判断 (指摘 3)

上記を設計書に反映してから実装に入ってほしい。特に指摘 1, 2, 4 は実装時に迷うポイントなので、プランナーかユーザーに確認して、設計書に追記すること。
