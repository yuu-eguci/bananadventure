# モーダル角丸をカード方針に合わせる設計ノート

## タスク概要
- 目的: 「やった！でもおなかいっぱいでもう今日は動けないのだ」などのメッセージモーダルの角丸を、カードと同じ丸み方針へ揃える
- 背景: カード角丸対応後、モーダルだけデフォルト角丸のままで統一感が不足

## 今回の要望（原文要約）
- モーダルの角丸が未対応だったので、前回同様にプランナー/レビュワー分業で修正したい

## 詳細設計（プランナー記入欄）
- 対象モーダル
  - `webapp/frontend-react/src/pages/HomePage.tsx` のメッセージ表示 `Dialog`
  - 対象メッセージ例: 「やった！でもおなかいっぱいでもう今日は動けないのだ」など、`responseText` を表示するダイアログ
- 変更方針
  - カード方針（`022`）と見た目をそろえるため、モーダル本体（`Dialog` の `Paper`）の角丸を通常カードと同じ `14px` に合わせる
  - 変更対象は角丸のみ。余白・文字サイズ・ボタン配置・アニメーションは変更しない
  - 角丸指定は `Dialog` 内部の `Paper` に限定し、ほかのコンポーネントへ影響を広げない
- 想定実装箇所
  - `HomePage.tsx` の `Dialog open={openDialog}` 定義箇所
  - MUI v7 の正しい指定を使い、`Dialog` に `slotProps={{ paper: { sx: { borderRadius: "14px" } } }}` を設定する
  - 既存で `slotProps.paper` がある場合は上書きせず、既存項目を残したまま `sx.borderRadius` だけ追記する
- 実装制約（角丸以外を変えないための明文化）
  - `Dialog` で変更してよいのは `Paper` の `borderRadius` だけに限定する
  - `sx` 内に `padding` / `margin` / `width` / `maxWidth` / `minWidth` / `boxShadow` / `backgroundColor` を追加・変更しない
  - `Dialog` の `open` / `onClose` / ボタン処理 / テキスト表示ロジックには触れない
- 確認方法
  - ゲームを進めてメッセージ `Dialog` を表示し、モーダル角丸がカード（`14px`）と同じ丸みに見えることを確認
  - PC 基準: ブラウザ幅 `1280px` 以上で確認し、カードとモーダルを見比べて角丸の大きさが同等に見えること
  - スマホ基準: 幅 `390px`（iPhone 12 相当）で確認し、モーダル四隅がつぶれず、カードと同じ `14px` の丸みとして視認できること
  - `OK` ボタン押下、モーダルの開閉、背景操作（`onClose`）が変更前と同じ挙動か確認
  - 差分レビュー時に、`Dialog` まわりで `borderRadius` 以外のスタイル差分がないことを確認

## レビュー（レビュワー記入欄）
全体的にはカード角丸 `022` 方針を踏襲した設計になっていて、問題ない。細かい確認事項をいくつか挙げる。

### 指摘事項
- **MUI v7 の指定方法について**
  - `Dialog` の `slotProps.paper.sx.borderRadius` という表記が登場しているが、MUI v7 では `slotProps={{ paper: { sx: { borderRadius: "14px" } } }}` の形式が正しい。この点を実装時に明示しておくと誤記を防げる
- **PC/スマホ相当幅の確認基準が曖昧**
  - 「違和感がないか」という目視確認だけでは実装者によって判断がブレる。PC 幅・スマホ幅それぞれで「角丸が 14px に見えるか」「カード UI と並べたときに統一感があるか」のように具体的な確認観点を追加すると、レビュー時にも判定しやすい
- **角丸以外のスタイル変更が混ざらないことを保証する手段**
  - セルフレビューで「角丸以外の差分が入っていないか確認する」と書いてあるが、具体的にどの差分を見ればいいかが書かれていない。実装メモに「`sx` の中に `borderRadius` だけを追加し、既存の `padding`・`margin` などには触れない」といった制約を明記しておくと安全
- **既存の `Paper` スタイル指定の有無を事前確認**
  - `HomePage.tsx` の `Dialog` に既に別の `Paper` スタイル指定（例: `maxWidth` や `elevation` など）が存在する場合、`borderRadius` を追加する際に上書きや競合が起きる可能性がある。実装前に既存 `Paper` 指定の有無を確認し、追記する形か新規追加かを明確にしておくとスムーズ

### 補足
- 角丸以外の変更を避ける方針は適切。カード方針と同じ `14px` を採用する判断も妥当
- 確認方法のチェックリスト化は良いアプローチ。実装後のセルフレビューと記録が期待できる

### 判定
上記の指摘を実装前に反映するか、実装時に意識すれば問題なく進められる設計なので、全体としては **LGTM**
