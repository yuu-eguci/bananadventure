# BGM 追加フィージビリティレポート

## タスク概要

React アプリに BGM を追加したい。
ユーザのフィードバック: 「BGM 追加希望 (流れる流れないの2通りじゃなくて、ユーザがポチッとしたら流れる、もう一回ポチッとしたら止まるみたいな)」

### 今回の目的

- React アプリへの BGM 追加が技術的に可能かどうかを調査する
- 可能であれば、どのように実装するかの方針をまとめる
- 実装は React (Vite + TypeScript + MUI) 上で行う

### 求められる UI 挙動

- ボタンを押すと BGM が流れ始める
- もう一度ボタンを押すと BGM が止まる (トグル)
- ループ再生 (ゲーム BGM なので無限ループが望ましい)

### 注意事項

- React への音楽追加は今回が初の試み
- まずはフィージビリティと実装アプローチのレポートを作成する段階
- 音楽ファイルは実際に mp3 が手元にある。それをプロジェクトに組み込んで流す想定

## 詳細設計（プランナー記入欄）
### 1. React で BGM を扱う技術的な可否
結論、**技術的に十分可能**です。  
今回ほしい挙動（ボタンで再生/停止トグル、ループ再生）は、React の `useState` + `useRef` とブラウザ標準 API で実現できます。

- トグル: `isPlaying` 状態で制御
- 再生/停止: `audio.play()` / `audio.pause()`
- ループ: `audio.loop = true`
- 音量: `audio.volume = 0.2` などで調整

注意点は Autoplay Policy だけなので、**ユーザ操作起点で再生開始**にすれば実運用できます。

### 2. 利用可能なアプローチ比較
| アプローチ | 向いているケース | メリット | デメリット |
|---|---|---|---|
| `HTMLAudioElement` (`new Audio()`) | BGM 1〜2本、シンプルトグル | 実装が最短、学習コスト低い、依存追加なし | 高度なミキシングやエフェクトは弱い |
| Web Audio API (`AudioContext`) | 複数音源、フェード、ミキシング | 音制御が柔軟（GainNode など） | 実装コスト高、Autoplay 対応も少し複雑 |
| 音声ライブラリ（例: howler.js） | 将来 SE/BGM 管理を拡張したい | クロスブラウザ差分を吸収しやすい | 依存追加、プロジェクト規模に対して過剰になりやすい |

### 3. 推奨アプローチと理由
今回は **`HTMLAudioElement` を推奨**します。

- 要件が「BGM の ON/OFF トグル + ループ」に限定されている
- 依存を増やさずに短く安全に導入できる
- 既存構成（React + TypeScript + MUI）に自然に乗る
- 将来複雑化した時だけ Web Audio API / ライブラリへ段階移行しやすい

つまり、最初は軽く始めて、必要になったら拡張する方針がちょうど良いです。

### 4. 推奨アプローチの実装イメージ（コードスケッチ）
`useBgmPlayer` フックで再生制御を 1 箇所にまとめる想定です。

```tsx
// src/hooks/useBgmPlayer.ts
import { useEffect, useRef, useState } from "react";

export function useBgmPlayer(src: string) {
  const audioRef = useRef<HTMLAudioElement | null>(null);
  const [isPlaying, setIsPlaying] = useState(false);
  const [error, setError] = useState<string | null>(null);

  useEffect(() => {
    const audio = new Audio(src);
    audio.loop = true;
    audio.volume = 0.25;
    audioRef.current = audio;

    return () => {
      audio.pause();
      audio.currentTime = 0;
      audioRef.current = null;
    };
  }, [src]);

  const toggle = async () => {
    if (!audioRef.current) return;
    try {
      if (isPlaying) {
        audioRef.current.pause();
        setIsPlaying(false);
        return;
      }
      await audioRef.current.play(); // ユーザ操作起点で呼ぶ
      setIsPlaying(true);
      setError(null);
    } catch {
      setError("音声再生がブラウザにブロックされました");
    }
  };

  return { isPlaying, toggle, error };
}
```

`HomePage` ではこんな感じでトグルボタンを置きます。

```tsx
const { isPlaying, toggle } = useBgmPlayer("/bgm/adventure-loop.mp3");

<Button variant="contained" onClick={toggle}>
  {isPlaying ? "BGM OFF" : "BGM ON"}
</Button>
```

### 5. 音声ファイルの配置場所
このプロジェクトは画像を `src/assets` と `public` の両方で扱っていますが、BGM はまず **`public/bgm/` 配置**を推奨します。

- 推奨配置: `webapp/frontend-react/public/bgm/adventure-loop.mp3`
- 参照方法: `"/bgm/adventure-loop.mp3"`

理由:
- `assetImageResolver` は画像専用の解決ロジックで、音声は対象外
- BGM はビルド時 import なしで固定パス参照できると運用がシンプル
- 音源差し替え時にコード変更が最小になる

### 6. ブラウザ自動再生ポリシー（Autoplay Policy）対応方針
基本方針は「**初回再生は必ずユーザ操作で開始**」です。

- 初期状態は `isPlaying = false`
- `useEffect` で自動 `play()` はしない
- ボタンクリック内で `play()` を呼ぶ
- `play()` の Promise reject（`NotAllowedError` 等）を捕捉して UI に案内を出す
- モバイル Safari 含め、ユーザ操作起点なら通る設計にする

### 7. 変更対象ファイルの一覧（実装時想定）
- 追加: `webapp/frontend-react/src/hooks/useBgmPlayer.ts`
- 追加: `webapp/frontend-react/src/components/BgmToggleButton.tsx`  
  （`HomePage` 直書きでも可。再利用性を考えるならコンポーネント化推奨）
- 変更: `webapp/frontend-react/src/pages/HomePage.tsx`  
  （BGM トグル UI 設置とフック呼び出し）
- 追加: `webapp/frontend-react/public/bgm/adventure-loop.mp3`
- 任意変更: `webapp/frontend-react/src/locales/ja/translation.json`
- 任意変更: `webapp/frontend-react/src/locales/en/translation.json`  
  （ボタン文言を i18n 管理する場合）

### 1. 技術選定の妥当性

**LGTM です**。今回のシンプルな要件 (トグル + ループ) には `HTMLAudioElement` がベスト。依存を増やさない判断も正しいし、段階的拡張の余地を残してるのも賢いです。

---

### 2. 実装イメージ (コードスケッチ) の正確さと抜け漏れ

#### 🟡 **指摘 1: `useEffect` の依存配列に `src` が含まれているけど、実際に `src` が変わる想定があるの？**

コードスケッチでは `useEffect` の依存配列に `[src]` が入ってるけど、`HomePage` では固定パス (`"/bgm/adventure-loop.mp3"`) を渡す想定だよね。
もし `src` が変わらないなら、空配列 `[]` の方が意図が明確になるし、再マウント時の挙動も予測しやすくなるよ。

**推奨**: `src` が固定なら `useEffect(() => { ... }, [])` にするか、あるいは「将来 BGM を切り替える想定があるなら、その旨を設計書に明記する」のがいいと思う。

#### 🟡 **指摘 2: エラーハンドリングが甘い可能性**

`catch` ブロックで `setError("音声再生がブラウザにブロックされました")` としてるけど、`play()` の失敗理由は他にもあるよ (ファイルが見つからない、デコードエラーなど)。
最低限 `error` を型チェックするか、ログ出力する方がデバッグしやすくなるよ。

**推奨**: `catch (err)` で `err instanceof DOMException` などをチェックして、`err.name === 'NotAllowedError'` だけ UI に出すか、それ以外は console に詳細を残すといいです。

#### 🟡 **指摘 3: `audio.currentTime = 0` のタイミング**

`useEffect` のクリーンアップで `audio.currentTime = 0` してるけど、これって必要かな？
`audio.pause()` だけで十分だし、次回再マウント時に新しい `Audio` インスタンスが作られるから、`currentTime` のリセットは不要だと思うよw

**推奨**: コードスケッチから削除するか、意図があるならコメントで明記するといいです。

---

### 3. ブラウザ Autoplay Policy 対応の十分さ

**ほぼ LGTM だけど、1 点追記してほしい**。

#### 🟡 **指摘 4: iOS Safari での「無音トラック」トリックは不要だけど、その判断を明記してほしい**

Autoplay Policy の対応として「ユーザ操作起点で `play()` を呼ぶ」は正しいけど、iOS Safari には「最初に無音トラックを再生しておく」みたいな古いテクニックもあるよね。
今回はそれを採用しない判断だと思うけど、その理由 (「ユーザ操作起点で十分」) を設計書に一言書いておくと、後で「なんで無音トラックやってないの？」って疑問に答えやすくなるよ。

**推奨**: 6 章に「iOS Safari 向けの無音トラック再生は不要と判断。理由は、ユーザ操作起点で `play()` を呼べば問題なく動作するため」みたいな一文を追加するといいです。

---

### 4. 音声ファイル配置方針の妥当性

**LGTM です**。`public/bgm/` に置く判断は正しいし、理由も明確です♡
`assetImageResolver` が画像専用って理解もバッチリだね。

### 5. 変更対象ファイルリストの網羅性

#### 🟡 **指摘 5: CI 設定 (`.github/workflows/ci.yml`) の確認が抜けてる**

`public/bgm/` に mp3 を追加する場合、CI のビルドで静的ファイルが正しくコピーされるか確認した方がいいよ。
特に Vite のビルド設定や、Docker イメージに `public/` が含まれるかのチェックが必要になるかもしれない。

**推奨**: 変更対象ファイルリストに「**確認**: `.github/workflows/ci.yml` (ビルド・静的ファイルのコピー確認)」を追加するといいです。

#### 🟡 **指摘 6: `BgmToggleButton.tsx` をコンポーネント化する判断基準が曖昧**

設計書では「`HomePage` 直書きでも可。再利用性を考えるならコンポーネント化推奨」って書いてあるけど、この判断基準が曖昧だよね。
今回は BGM ボタンは `HomePage` にしか置かない想定なのか、将来他のページにも置く可能性があるのか、その見通しを明記してほしいです。

**推奨**: 「当面は `HomePage` のみに配置する想定。将来他ページにも配置する場合はコンポーネント化を検討する」みたいな方針を 7 章に書くといいよ。
