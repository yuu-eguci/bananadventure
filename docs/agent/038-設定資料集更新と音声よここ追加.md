# 設定資料集更新と音声よここ追加

## タスク概要
- 目的: `/lore` の設定資料集を、内容更新・キャラクター表示・音声再生機能の3点で拡張する
- 背景: 既存の `037-設定資料集ページ追加` は単一QAのみで、今回の要望（Q/A拡張・よここ表示・音声ON/OFF）を満たしていない

## 今回の要望（原文要約）
- 設計ノート 037 相当: 内容追加
  - サブタイトルを `ばななちゃんのひみつ、教えちゃうよ！` から `ばななちゃんのひみつ、よここさんが教えちゃうよ！` へ変更
  - 既存の「このゲームにはどんな曲がある？」QAを置き換え、Q1〜Q5の新しいQAセットを追加
  - Q3の回答末尾に YouTube 風アイコン付きリンクを表示し、`https://www.youtube.com/@うみにょ·uminyo` へ遷移させる
- 設計ノート 038 相当: キャラクター「よここ」追加
  - A表示エリアで、回答テキスト左側に `001-yokoko.webp` を表示する
  - 回答をキャラクターが話している見せ方にする
- 設計ノート 039 相当: 音声追加
  - `/lore` に `音声 ON/OFF` 機能を導入する（`/` の BGM ON/OFF と同様の操作感）
  - QをタップしてAが開いたタイミングで、対応するQ音声→A音声を順番に再生する
  - 再生順は以下:
    - Q1: `001_Q1.mp3` -> `002_A1.mp3`
    - Q2: `003_Q2.mp3` -> `004_A2.mp3`
    - Q3: `005_Q3.mp3` -> `006_A3.mp3`
    - Q4: `007_Q4.mp3` -> `008_A4.mp3`
    - Q5: `009_Q5.mp3` -> `010_A5.mp3`

## 詳細設計（プランナー記入欄）

### 1. 今回の更新方針
- 既存の `/lore` は単一QAのシンプルな Accordion ページなので、今回もベースは `LorePage.tsx` のまま活かしつつ、`QAデータの拡張` `A表示レイアウト変更` `音声制御追加` の3点を足していく
- UI は 037 のトンマナを崩さない。つまり、`primary.main` の黄色ヘッダ、丸みのある Accordion、白ベースの本文、下部の「ゲームに戻る」ボタンは維持する
- 今回の変更対象は実装時には主に `src/pages/LorePage.tsx` `src/locales/ja/translation.json` `src/locales/en/translation.json` `src/hooks/useLoreVoicePlayer.ts` を想定する。設計段階ではこの責務分割で固める

### 2. QAデータの持ち方
- `LorePage.tsx` 内の `loreItems` は単一要素配列から5件配列へ差し替える
- 各要素は「表示文言」と「音声ファイル」と「任意リンク」をまとめて持つ。質問・回答本文そのものは i18n に寄せ、ページ側はキーとメタ情報だけ持つ
- データ形は以下を想定する

```ts

    answerKey: "lore.q1.answer",

    answerVoiceSrc: "/voice/002_A1.mp3",

    id: "q2",
    questionKey: "lore.q2.question",
    answerKey: "lore.q2.answer",

    answerVoiceSrc: "/voice/004_A2.mp3",

    id: "q3",
    questionKey: "lore.q3.question",
    answerKey: "lore.q3.answer",
    questionVoiceSrc: "/voice/005_Q3.mp3",
    answerVoiceSrc: "/voice/006_A3.mp3",
    link: {
      href: "https://www.youtube.com/@うみにょ·uminyo",
      labelKey: "lore.q3.youtubeLinkLabel",

    id: "q4",
    questionKey: "lore.q4.question",

```

- 既存の `lore.bgm.question` `lore.bgm.answer` は今回の LorePage からは使わない。実装では Q1〜Q5 の確定文言へ置き換える
- Q1〜Q5 の本文は、要望元で受け取っている確定稿を各翻訳キーへそのまま入れる。ページ内にベタ書きしない

### 3. ページ状態の設計
- Accordion は今の「各項目が勝手に開閉する」形ではなく、`expandedItemId` を持つ制御コンポーネントに変える
- 状態は最低限これを持つ
  - `expandedItemId: "q1" | "q2" | "q3" | "q4" | "q5" | false`
  - `isVoiceEnabled: boolean`
  - `playingItemId: "q1" | "q2" | "q3" | "q4" | "q5" | null`
- 動作ルールは以下で固定する
  - 閉じているQをタップ: そのQを開く
  - 開いている同じQをもう一度タップ: 閉じる。再生中なら音声も止める
  - 別Qをタップ: 今の再生を止めてから新しいQを開き、そのQの音声シーケンスへ切り替える
  - `音声 OFF` のときは開閉だけ行い、音声は再生しない

### 4. レイアウト設計
- ページ上部は `タイトル/サブタイトル` と `音声 ON/OFF ボタン` を同じ行グループに置く
- ただしスマホでは横幅が足りないので、`Stack` か `Box` のレスポンシブ指定で縦並びに落とす
- 構成イメージはこんな感じ

```text
Box (maxWidth: 960, mx: auto, py: 4, px: { xs: 2, sm: 3 })
  ├─ Stack (xs: column, sm: row, justifyContent: space-between, gap: 2)
  │   ├─ Box
  │   │   ├─ Typography h4: 設定資料集
  │   │   └─ Typography body2: ばななちゃんのひみつ、よここさんが教えちゃうよ！
  │   └─ Button: 音声 ON/OFF
  ├─ Box (mt: 3)
  │   └─ loreItems.map => Accordion
  │       ├─ AccordionSummary: 質問
  │       └─ AccordionDetails
  │           └─ Box (display: flex, gap: 2, alignItems: flex-start)
  │               ├─ Box component="img": よここ画像
  │               └─ Box: 回答吹き出しエリア
  │                   ├─ Typography: 回答本文
  │                   └─ Q3 のみ YouTube リンク
  └─ Box (mt: 4, textAlign: center)
      └─ Button: ゲームに戻る
```

- Accordion の見た目は 037 レビューで固めた方針を引き続き採用する
  - `disableGutters`
  - `elevation={0}`
  - `borderRadius: "14px !important"`
  - `overflow: "hidden"`
  - `&:before: { display: "none" }`
  - `mb: 1.5`
- 質問ヘッダは引き続き `primary.main` を使う。ここを別デザインに変えない
- Aエリアは「よここが答えている」見せ方に寄せるため、回答本文をただの `Typography` 1枚で出さず、白背景の吹き出し風 Box or Paper に入れる
- `001-yokoko.webp` は回答本文の左に固定で出す。実装では `src/assets/character-image/001-yokoko.webp` を import して使う
- スマホでも「左にキャラ、右に吹き出し」の関係は維持する。ただし画像サイズは `xs` では小さめにしてレイアウト崩れを防ぐ

### 5. Q3 の YouTube アイコン付きリンク
- Q3 だけは回答文末の下にリンク行を追加する
- リンクは通常テキストではなく、MUI の `Link` か `ButtonBase` に `YouTube` アイコンを添えた見た目にする
- 想定仕様
  - アイコン: `@mui/icons-material/YouTube`
  - 表示位置: Q3 回答テキストの直下、`mt: 1.5` 程度
  - 遷移先: `https://www.youtube.com/@うみにょ·uminyo`
  - `target="_blank"` `rel="noreferrer"` を付与
- クリック可能なのは Q3 のみで、Q1/Q2/Q4/Q5 には余計なプレースホルダを出さない
- リンク文言は実装で固定文字列にせず、`lore.q3.youtubeLinkLabel` の翻訳キーから取る

### 6. 音声 ON/OFF トグル
- `/` の BGM ボタンと同じく、トグル式の1ボタンで状態を切り替える
- ただし LorePage には大きなメイン画像がないので、ページ右上固定ではなくヘッダ右側に置く。見た目だけ `/` と揃え、レイアウト都合で配置は LorePage 用に自然な位置へ寄せる
- ボタン仕様
  - ラベル: `音声 ON` / `音声 OFF`
  - アイコン: `VolumeUpRounded` / `VolumeOffRounded`
  - ON 時は `color="success"` 寄り、OFF 時はグレー寄りで HomePage の BGM ボタンと同じ操作感にする
  - `aria-label` も ON/OFF に応じて切り替える
- 初期値は `OFF` にする
  - 理由1: `/` の BGM もユーザー操作で明示的に再生開始する設計だから、LorePage も揃えたほうが自然
  - 理由2: ブラウザの自動再生制限を避けやすい
- `音声 OFF` に切り替えた瞬間、再生中の音声があれば即停止する。Accordion の開閉状態は維持する

### 7. 音声再生の責務分割
- 音声制御は `LorePage.tsx` に全部書き込まず、`useLoreVoicePlayer.ts` のような専用 hook に切り出す
- hook の責務は以下
  - 単一の `HTMLAudioElement` を保持する
  - Q音声→A音声の2本を順番に再生する
  - 再生停止、再生中アイテム管理、連打時のキャンセルを担う
  - ページアンマウント時に確実に停止する
- hook から返すものはこのくらいで十分

```ts
const {
  isVoiceEnabled,
  setVoiceEnabled,
  playingItemId,
  playSequence,
  stop,
} = useLoreVoicePlayer();
```

### 8. 音声再生フロー
- Qタップ時の処理順はこうする
  1. タップされたQが現在開いているQと同じなら `expandedItemId` を `false` にする
  2. 同じQを閉じるケースなら `stop()` を呼んで終了
  3. 別Qを開くケースなら、まず今の再生を `stop()` する
  4. `expandedItemId` を新しいQへ更新する
  5. `isVoiceEnabled === true` のときだけ `playSequence(item)` を呼ぶ
- `playSequence(item)` の中では `questionVoiceSrc` を再生し、終了イベントを受けたら `answerVoiceSrc` を再生する
- A音声まで終わったら `playingItemId` を `null` に戻す
- 開いているQを変えずに `音声 ON` にしただけでは自動再生しない。再生開始のトリガーはあくまで「Qをタップしたとき」に限定する
  - これで意図しない自動再生を防げる

### 9. 連打時・別Q切替時の競合対策
- ここは今回いちばん大事なので、実装方針を明確に固定する
- `requestIdRef` のような単調増加の識別子を hook 内に持ち、再生要求ごとに採番する
- 新しいQが押されたり OFF になった時点で `requestIdRef` を進め、古い再生要求を無効化する
- `playSequence` と `playSingleClip` は毎回「自分の requestId がまだ最新か」を確認してから次へ進む
- 停止処理では必ず以下を行う
  - `audio.pause()`
  - `audio.currentTime = 0`
  - `playingItemId = null`
  - 旧 requestId を無効化
- これで防ぐこと
  - 同じQを連打して Q音声が二重に鳴る
  - Q1 の A音声再生中に Q2 を押したら Q1 と Q2 が同時に鳴る
  - Q3 を開いた直後に OFF にしたのに、遅れて A3 が鳴り始める
- 音声要素は毎回複数生成せず、hook 内の単一 `Audio` を使い回す方針にする。複数インスタンスを並行で持たないことで競合点を減らす

### 10. 翻訳キーの更新方針
- 既存キーの更新
  - `lore.subtitle` を新文言へ差し替える
- 新規キーの追加

  - `lore.q1.answer`
  - `lore.q2.question`
  - `lore.q2.answer`
  - `lore.q3.question`
  - `lore.q3.answer`
  - `lore.q3.youtubeLinkLabel`

  - `lore.q5.question`

  - `lore.voiceToggle.ariaOn`

- `ja` と `en` の両方に同じキーを追加する
- 日本語文言は要望元の確定稿をそのまま反映する。英語は暫定英訳でよいが、キー構造は日本語と完全一致させる

## 実装メモ（プランナー記入欄）
- `001-yokoko.webp` は `public` ではなく `src/assets/character-image` 配下にあるので、`import yokokoImage from "@/assets/character-image/001-yokoko.webp"` のように import して使う前提で実装する
- 音声ファイルは `public/voice` 配下にそろっているので、再生パスは `/voice/001_Q1.mp3` のような絶対パスで統一する
- Q3 の回答本文にリンク文字列を埋め込むのではなく、「回答本文」と「リンク行」を別要素で描画する。これなら翻訳文言の差し替えと見た目調整を分離できる
- 音声再生の実装では `ended` イベントの都度リスナーを張りっぱなしにしない。都度解除するか、Promise 化したヘルパーに閉じ込める
- LorePage を離れるときに音声が残らないよう、hook の `useEffect` cleanup で必ず停止する
- `音声 ON/OFF` の見た目は HomePage の BGM ボタンに寄せるが、LorePage で固定配置にはしない。既存本文や戻るボタンに被せないことを優先する

## レビュー (レビュワー追記)

### 指摘事項

- **2. QAデータの持ち方**: `as const` を使っているけど、実際には各 QA アイテムの型定義を明示したほうがいい。 `link` が optional なのに型アサーションだけだと TypeScript の恩恵が薄い。 `LoreItem` みたいな interface を切ってから使うほうが安全
- **3. ページ状態の設計**: `playingItemId` と `expandedItemId` を別々に持つのは OK だけど、実装時に「Aが開いているのに playingItemId が null」みたいな中間状態が発生しないようにフロー図を補足しておくといい。特に連打や OFF 切替時の状態遷移が曖昧になりやすい
- **4. レイアウト設計**: 「回答吹き出しエリア」の具体的な見た目 (角丸、影、背景色) が書かれていない。よここ画像とのバランスやスマホ時の縮小率も実装メモに入れておかないと、実装者が迷う
- **5. Q3 の YouTube アイコン付きリンク**: `ButtonBase` を使うなら `role="link"` と `href` をちゃんと設定する必要があるけど、`ButtonBase` には `href` がないから実装時に `Link` か `Button` + `onClick` のどちらかに統一する方針を明記したほうがいい
- **7. 音声再生の責務分割**: `useLoreVoicePlayer` の戻り値に `playSequence` があるけど、引数の型が書かれていない。 `LoreItem` 全体を渡すのか、音声パスだけ渡すのか実装メモに書いておかないと実装時に揺れる
- **9. 連打時・別Q切替時の競合対策**: `requestIdRef` の採番タイミングと無効化タイミングをフローチャートか疑似コードで補足しないと、実装時に「どこで increment するか」で迷う可能性がある。特に `playSequence` 開始前と `playSingleClip` 開始前のどちらで increment するかが曖昧
- **10. 翻訳キーの更新方針**: `lore.bgm.question` と `lore.bgm.answer` を「今回の LorePage からは使わない」と書いているけど、削除するのか残すのか明記されていない。他ページで使っているなら残すべきだし、使っていないなら削除してキーを整理すべき
- **11. 影響ファイルの想定**: `LorePage.tsx` の変更内容が「QA5件化、よここレイアウト、Q3リンク、音声トグル、制御された Accordion 化」と盛りだくさんすぎる。実装時に差分が大きくなりすぎるので、「どの順番で実装するか」の優先順位を補足しておくといい
- **全体**: `001-yokoko.webp` のサイズ (width/height) やスマホ時の縮小比率が書かれていない。アセットの実寸が不明だと、実装時に「どれくらいのサイズで表示するか」で迷う。実装メモに `xs: 60px, sm: 80px` みたいな具体値を入れておくといい

### 良い点

- QAデータと音声パスの紐付けをひとつの配列で管理する方針は明快で、実装時の見通しが良い
- Accordion の見た目を 037 から引き継ぐ方針を明記しているので、デザインの一貫性が保たれる
- 音声 ON/OFF の初期値を OFF にする理由がちゃんと書かれていて、仕様判断の根拠が明確
- `useLoreVoicePlayer` に責務を切り出す方針は正しい。ページコンポーネントが肥大化しないで済む
- 連打対策に `requestIdRef` を使う方針はベストプラクティス。ただし実装時の詳細フローを補足する必要あり
- 実装メモで `public` vs `src/assets` の配置を明記しているのは親切

### 総評

設計の大枠は問題ないけど、実装時に迷いやすい部分 (型定義、リンクの実装方法、画像サイズ、翻訳キーの削除可否、実装順序) の詳細が抜けている。あと連打対策の `requestIdRef` フローをもうちょっと具体的に書かないと、実装者が自己流で書いて結局バグる可能性がある。ここを補足すれば LGTM だけど、今の状態だと実装中に仕様確認が何度か発生しそうｗ
